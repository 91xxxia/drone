# 无人船虚拟仿真系统开发实施计划（基于 RuoYi-Vue3 + Spring Boot）

> 日期：2026-04-26  
> 依据文档：`doc/无人船虚拟仿真系统功能需求书.md`、`sql/drone.sql`

---

## 1. 交叉分析结论（需求书 vs 现有 SQL）

### 1.1 当前已具备能力（可直接复用）
- **RBAC 基础能力完整**：已有 `sys_user/sys_role/sys_menu/sys_role_menu/sys_user_role`，满足管理员/普通用户权限分层。
- **系统日志能力完整**：已有 `sys_oper_log`、`sys_logininfor`，可承接关键业务操作审计。
- **字典能力完整**：已有 `sys_dict_type/sys_dict_data`，可管理设备状态、任务状态等枚举。
- **代码生成器可用**：已有 `gen_table/gen_table_column`，可用于快速生成基础 CRUD。
- **定时调度基础具备**：Quartz 表齐全，后续如要做任务进度定时推进可扩展使用。

### 1.2 关键缺口（必须新增）
- 当前 SQL 中**无无人船业务表**（设备、航线、任务、结果等均不存在）。
- 前后端代码中未发现 `drone/usv/patrol/route/mission/result` 业务模块。
- 普通角色（`role_key=common`）在初始化 SQL 中被赋予了大量系统菜单与按钮权限，不符合“普通用户只读+少量操作”目标，需按需求重构权限集。

### 1.3 结论
本项目应按“**在 RuoYi 基座上新增完整业务域**”方式推进：
1. 先落地数据模型与字典；
2. 再完成后端领域服务与状态机；
3. 最后完成 Vue3 页面、权限与联调验收。

## 2. 目标架构与模块划分

## 2.1 后端模块建议（保持 RuoYi 标准分层）
- **领域对象**：放在 `ruoyi-system/src/main/java/com/ruoyi/system/domain`。

- **Service 接口/实现**：放在 `.../service`、`.../service/impl`。
- **Controller**：放在 `ruoyi-admin/src/main/java/com/ruoyi/web/controller/system`（或新建 `controller/drone` 包）。

## 2.2 前端模块建议（Vue3 规范）
- **API**：`RuoYi-Vue3-master/src/api/drone/*.js`
- **页面**：`RuoYi-Vue3-master/src/views/drone/*`

- **路由入口**：通过后端菜单动态路由，不手写静态业务路由。

## 3. 数据库实施方案（第一优先级）

## 3.1 建议新增核心表

1. `biz_usv_device`（设备表）
- 主键：`device_id`
- 唯一：`device_code`（设备编号）
- 关键字段：`device_name/model/endurance_minutes/camera_spec/owner_name/status/remark`
- 状态建议：`normal/maintenance/offline/in_task/scrapped`

2. `biz_patrol_route`（航线主表）
- 主键：`route_id`
- 唯一：`route_name`
- 字段：`est_duration_minutes/altitude/remark`

3. `biz_patrol_route_point`（航线点位表）
- 主键：`point_id`
- 外键：`route_id`
- 字段：`seq/lng/lat`
- 约束：`(route_id, seq)` 唯一

4. `biz_patrol_task`（任务表）
- 主键：`task_id`
- 唯一：`task_code`
- 外键：`device_id/route_id`
- 字段：`task_name/executor/planned_start_time/task_desc/status/progress`
- 状态：`pending/running/completed/canceled`

5. `biz_patrol_task_log`（任务状态流转日志）
- 主键：`log_id`
- 外键：`task_id`
- 字段：`from_status/to_status/operate_by/operate_time/remark`

6. `biz_patrol_result`（结果表）
- 主键：`result_id`
- 唯一：`result_code`
- 外键：`task_id`（建议唯一，确保任务只生成一条主结果）
- 字段：`overview/findings/handling/remark/duration_minutes/completed_time/executor`

7. `biz_patrol_result_media`（结果图片/识别信息，可选扩展）
- 主键：`media_id`
- 外键：`result_id`
- 字段：`file_url/ai_tag/bbox_json`

## 3.2 索引与约束策略
- 任务查询高频索引：`idx_task_status`、`idx_task_executor`、`idx_task_start_time`
- 结果查询高频索引：`idx_result_code`、`idx_result_completed_time`
- 删除约束：
  - 已绑定任务的设备、航线不可删（业务层拦截 + 数据层约束）
  - 已完成任务不可改绑定关系

## 3.3 字典落地（`sys_dict_type/sys_dict_data`）
- `drone_device_status`：正常、维修中、离线、任务中、已报废
- `drone_task_status`：待执行、执行中、已完成、已取消
- `drone_result_level`（可选）：正常、轻微异常、严重异常

## 4. 后端实施计划（按迭代）

## 4.1 迭代 A：基础 CRUD（2~3 天）
- 用代码生成器生成：设备、航线主表、任务、结果基础 CRUD。
- 手工补齐：
  - 航线点位子表的增删改查（生成器对子表支持有限，建议手写）。
  - 业务唯一性校验（设备编号、结果编号等）。

## 4.2 迭代 B：核心业务规则（3~4 天）
- 实现任务状态机：
  - `pending -> running`
  - `running -> completed`
  - `pending/running -> canceled`
- 联动规则：
  - 任务开始：设备状态改为 `in_task`
  - 任务完成/取消：设备状态恢复 `normal`
  - 任务完成自动生成 `biz_patrol_result`
- 防御式校验：
  - 任务开始时间不得早于当前时间
  - 设备状态必须为 `normal` 才允许创建任务绑定
  - 同一航线不可绑定到同一“未完成”冲突任务

## 4.3 迭代 C：详情与导出（2 天）
- 设备详情：关联历史任务列表。
- 航线详情：回显点位与历史任务。
- 结果导出：复用 RuoYi Excel 注解 + `ExcelUtil` 实现结果列表导出。

## 4.4 迭代 D：日志与审计（1 天）
- 对关键接口加 `@Log`：设备增改删、任务状态切换、结果编辑/删除/导出。
- 保证操作日志与需求书一致可追溯。

## 6. 权限与菜单实施计划（RuoYi RBAC）

## 6.1 菜单目录结构建议
- 一级目录：`无人船巡防`
  - `设备管理`
  - `航线管理`
  - `任务管理`
  - `结果管理`
  - `数据概览`

## 6.2 权限标识建议
- 设备：`drone:device:list|query|add|edit|remove|export`
- 航线：`drone:route:list|query|add|edit|remove`
- 任务：`drone:task:list|query|add|start|finish|cancel|remove`
- 结果：`drone:result:list|query|edit|remove|export|detail`

## 6.3 角色权限策略
- 管理员：全权限。
- 普通用户：`list/query/detail` + 个人中心改密；禁用新增/编辑/删除/状态切换。
- 对当前 `role_key=common` 建议做一次“权限瘦身”SQL 修正，避免越权。

## 7. 代码生成器使用策略（降本增效）

## 7.1 推荐“生成 + 手工增强”边界
- **生成器适用**：单表标准 CRUD（设备、任务、结果基础）。
- **手工实现**：
  - 航线点位子表（主子表联动）
  - 任务状态机与设备状态联动
  - 结果自动生成
  - 地图相关 DTO（点位数组）

## 7.2 生成器配置建议
- `tpl_category=crud`
- 查询字段：名称、状态、时间范围
- 导出字段：结果管理的关键归档字段
- 前端模板类型：`element-plus`

## 10. 风险与应对

- **地图 API Key 风险**：提前申请高德 Key，封装环境变量，避免硬编码。
- **状态一致性风险**：任务与设备状态更新需同事务，避免脏状态。
- **权限遗漏风险**：接口层加 `@PreAuthorize`，前端按钮仅作展示控制。
- **导出性能风险**：结果导出默认按筛选条件分页/限流，防止一次性全量导出。
